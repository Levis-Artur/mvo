# Codex Development Guide

Version: 1.0

---

# Purpose

Цей документ визначає правила роботи Codex із проєктом MVO Inventory Management System.

Будь-яке завдання повинно виконуватися відповідно до цього документа.

---

# Before Starting Any Task

Перед початком будь-якої роботи необхідно ознайомитися з:

1. PROJECT_OVERVIEW.md
2. ARCHITECTURE.md
3. BUSINESS_RULES.md
4. SECURITY.md
5. ROADMAP.md

Лише після цього можна приступати до виконання задачі.

---

# General Rules

Codex не повинен змінювати архітектуру проєкту без окремого завдання.

Якщо запропоноване рішення суперечить існуючій архітектурі — необхідно спочатку пояснити проблему.

---

# Forbidden

Без окремої вказівки заборонено:

- змінювати структуру бази даних;
- перейменовувати API endpoint;
- змінювати URL;
- змінювати ролі користувачів;
- змінювати систему авторизації;
- змінювати бізнес-логіку;
- змінювати дизайн застосунку;
- видаляти існуючі можливості.

---

# Allowed

Без додаткового погодження дозволяється:

- рефакторинг;
- оптимізація;
- покращення читабельності;
- виправлення багів;
- написання тестів;
- очищення коду;
- видалення dead code.

---

# Architecture

Не допускається:

- бізнес-логіка у React-компонентах;
- SQL у контролерах;
- дублювання бізнес-логіки;
- доступ Frontend до БД;
- приховані API.

---

# Backend Rules

Кожен новий модуль повинен містити:

- Controller
- Service
- DTO
- Tests

За потреби:

- Guards
- Middleware
- Validators

---

# Frontend Rules

Кожен компонент повинен мати лише одну відповідальність.

Великі компоненти повинні розділятися.

Максимальний рекомендований розмір одного React-компонента — приблизно 400 рядків.

---

# API Rules

Усі нові endpoint повинні:

- використовувати DTO;
- проходити ValidationPipe;
- повертати стандартизовані помилки;
- використовувати Request ID;
- мати зрозумілі HTTP-коди.

---

# Database Rules

Усі зміни БД виконуються лише через:

Prisma Migration

Заборонено:

ручне редагування production database.

---

# Git Rules

Одна задача = один логічний commit.

Приклади:

feat:

fix:

refactor:

docs:

test:

chore:

---

# Testing

Після завершення будь-якої задачі необхідно виконати:

npm run lint

npm run build

npm test --workspace apps/api

npm test --workspace apps/web

Жодна задача не вважається завершеною, якщо хоча б один тест не проходить.

---

# Documentation

Якщо змінюється:

- архітектура;
- бізнес-правила;
- API;
- ролі;
- структура БД;

необхідно оновити відповідний документ у директорії docs.

---

# Pull Request Checklist

Перед завершенням роботи необхідно перевірити:

- lint
- build
- tests
- документацію
- Git diff

---

# Code Style

Перевага надається:

- простому коду;
- читабельності;
- явній логіці;
- невеликим функціям;
- модульності.

Складні рішення необхідно коментувати.

---

# Performance

Не оптимізувати передчасно.

Спочатку:

- правильна архітектура;
- коректність;
- тести.

Лише після цього оптимізація.

---

# Security

Жодне рішення не повинно:

- знижувати рівень безпеки;
- відкривати доступ до даних;
- обходити перевірку ролей;
- логувати секрети.

---

# If Something Is Unclear

Якщо вимоги неоднозначні —

не вигадувати рішення.

Спочатку запропонувати варіанти та пояснити їхні плюси і мінуси.
