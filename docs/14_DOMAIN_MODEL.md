Version: 1.0

# Purpose

Цей документ визначає бізнес-сутності системи MVO Inventory Management System.

Він є єдиним джерелом істини щодо значення основних понять.

Жоден розробник не повинен трактувати терміни інакше, ніж описано тут.

---

# Organization

Організація — верхній рівень структури.

Приклад:

Департамент патрульної поліції.

---

# Regional Division

Територіальний підрозділ.

Наприклад:

УПП Волинської області.

Саме на цьому рівні працює більшість адміністраторів.

---

# Structural Unit

Внутрішній підрозділ.

Наприклад:

- ТОР
- Рота №1
- Рота №2
- Відділ логістики

Користувач завжди належить одному структурному підрозділу.

---

# User

Користувач системи.

Має:

- логін;
- пароль;
- роль;
- прив'язку до підрозділу;
- прив'язку до МВО (за потреби).

---

# Materially Responsible Person (MVO)

Матеріально відповідальна особа.

Саме МВО відповідає за збереження майна.

Кожен МВО має власні залишки.

---

# Inventory Item

Одиниця обліку.

Може бути:

- кількісною;
- індивідуальною.

---

# Nomenclature

Довідник типів майна.

Приклади:

- бронежилет;
- каска;
- ноутбук;
- автомобіль;
- принтер.

Номенклатура не містить залишків.

# Inventory Instance

Конкретний екземпляр майна.

Використовується для майна з:

- серійним номером;
- інвентарним номером;
- VIN.

Приклади:

ноутбук

радіостанція

автомобіль

# Stock

`StockBalance` — прямий залишок майна accounting owner, який фізично перебуває
у нього. Він не включає чуже майно, закріплене за цим МВО.

`CustodyBalance` — поточна кількість майна одного accounting owner, закріплена
за іншим custodian. Accounting owner і custodian не можуть збігатися.

- `direct quantity` — `StockBalance.quantity`;
- `assigned quantity` — сума `CustodyBalance` accounting owner;
- `assigned to me` — custody, де МВО є custodian; це не його own balance;
- `total accounted quantity` — direct + assigned to others;
- `issued quantity` — остаточно вибула й не входить до поточного стану.

---

# Stock Movement

Будь-яка зміна залишку.

Не існує зміни залишку без відповідної операції.

---

# Business Document

Документ, який є підставою для зміни залишків.

Саме документ породжує операції.

Не навпаки.

---

# Receipt

Документ надходження.

Збільшує залишки.

---

# Transfer

У UI — передача майна новому фактичному утримувачу. Нові документи мають тип
`ASSIGNMENT`; accounting owner не змінюється, а own balance одержувача не
збільшується. `TRANSFER` — назва legacy-типу старої balance-to-balance моделі;
такі документи зберігаються як read-only історія.

---

# Issue

Остаточне вибуття з `DIRECT` або `ASSIGNED`. Проведення зменшує total accounted
accounting owner і вимагає фото/скан накладної. Скасування повертає кількість у
точний source bucket через reversal.

---

# Return

Документ повернення.

Збільшує залишок.

---

# Write-off

Документ списання.

Безповоротно списує майно.

---

# Transaction

Факт виконання документа.

Саме Transaction змінює залишки.

Документ без виконання не впливає на систему.

---

# Audit Event

Будь-яка важлива дія користувача.

Приклади:

- логін;
- імпорт;
- зміна пароля;
- створення документа;
- підтвердження передачі.

---

# Role

Набір дозволених можливостей.

Роль визначає:

що користувач може бачити;

що може змінювати;

над якими даними.

# Session

Поточна автентифікована сесія користувача.

---

# Attachment

# Import

Імпорт ніколи не використовується для передач між МВО.

---

# Report

Документ, сформований системою.

Не впливає на залишки.

---

# Notification

Повідомлення користувачу.

Не змінює бізнес-стан системи.

# Golden Rule

У системі змінюється не залишок.

У системі виконується документ.

Саме документ створює операцію.

Операція змінює залишок.

Цей принцип є фундаментом усієї системи.

## StockDocument

Універсальний документ руху майна. Тип `TRANSFER` передає майно між двома МВО,
тип `ISSUE` видає майно зовнішньому одержувачу. Статуси: `DRAFT`, `POSTED`,
`CANCELLED`. Рядки документа зберігаються у `StockDocumentLine`.
