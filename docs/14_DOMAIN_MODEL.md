# Domain Model

Version: 1.0

---

# Purpose

Цей документ визначає бізнес-сутності системи MVO Inventory Management System.

Він є єдиним джерелом істини щодо значення основних понять.

Жоден розробник не повинен трактувати терміни інакше, ніж описано тут.

---

# Organization

Організація — верхній рівень структури.

Приклад:

Департамент патрульної поліції.

---

# Regional Division

Територіальний підрозділ.

Наприклад:

УПП Волинської області.

Саме на цьому рівні працює більшість адміністраторів.

---

# Structural Unit

Внутрішній підрозділ.

Наприклад:

- ТОР
- Рота №1
- Рота №2
- Відділ логістики

Користувач завжди належить одному структурному підрозділу.

---

# User

Користувач системи.

Має:

- логін;
- пароль;
- роль;
- прив'язку до підрозділу;
- прив'язку до МВО (за потреби).

---

# Materially Responsible Person (MVO)

Матеріально відповідальна особа.

Саме МВО відповідає за збереження майна.

Кожен МВО має власні залишки.

---

# Inventory Item

Одиниця обліку.

Може бути:

- кількісною;
- індивідуальною.

---

# Nomenclature

Довідник типів майна.

Приклади:

- бронежилет;
- каска;
- ноутбук;
- автомобіль;
- принтер.

Номенклатура не містить залишків.

---

# Inventory Instance

Конкретний екземпляр майна.

Використовується для майна з:

- серійним номером;
- інвентарним номером;
- VIN.

Приклади:

ноутбук

радіостанція

автомобіль

---

# Stock

`StockBalance` — єдиний актуальний залишок номенклатури у конкретного МВО.
Кількість збільшується офіційним імпортом і зменшується проведеною передачею
або видачею. Від’ємний залишок заборонений.

`CustodyBalance` — legacy archive старої owner/custody-моделі. Нові операції не
створюють і не змінюють ці записи. Вони можуть читатися окремим audit adapter,
але не входять до поточних підсумків.

---

# Stock Movement

Будь-яка зміна залишку.

Не існує зміни залишку без відповідної операції.

---

# Business Document

Документ, який є підставою для зміни залишків.

Саме документ породжує операції.

Не навпаки.

---

# Receipt

Документ надходження.

Збільшує залишки.

---

# Transfer

У UI — «Передача», у новій моделі — `MVO_TRANSFER`. Проведення зменшує прямий
`StockBalance` відправника та не змінює `StockBalance` одержувача. Майбутній
прихід одержувачу можливий лише незалежним CSV-імпортом.

`TRANSFER` і `ASSIGNMENT` — legacy-типи. Вони зберігаються як read-only історія
та не можуть використовуватися для нових чернеток, posting або cancellation.

---

# Issue

Остаточне вибуття з прямого `StockBalance` МВО. Проведення створює `ISSUE_OUT`,
вимагає фото/скан накладної та зменшує тільки джерело. Скасування створює
`ISSUE_REVERSAL` і повертає кількість у той самий прямий залишок.

---

# Return

Документ повернення.

Збільшує залишок.

---

# Write-off

Документ списання.

Безповоротно списує майно.

---

# Transaction

Факт виконання документа.

Саме Transaction змінює залишки.

Документ без виконання не впливає на систему.

---

# Audit Event

Будь-яка важлива дія користувача.

Приклади:

- логін;
- імпорт;
- зміна пароля;
- створення документа;
- підтвердження передачі.

---

# Role

Набір дозволених можливостей.

Роль визначає:

що користувач може бачити;

що може змінювати;

над якими даними.

---

# Permission

Конкретне право.

Наприклад:

Create Transfer

Approve Receipt

Manage Users

View Reports

---

# Session

Поточна автентифікована сесія користувача.

---

# Attachment

Файл, прикріплений до документа.

Наприклад:

- PDF;
- фото;
- акт;
- накладна.

---

# Import

Операція масового завантаження даних.

Імпорт ніколи не використовується для передач між МВО.

---

# Report

Документ, сформований системою.

Не впливає на залишки.

---

# Notification

Повідомлення користувачу.

Не змінює бізнес-стан системи.

---

# Golden Rule

У системі змінюється не залишок.

У системі виконується документ.

Саме документ створює операцію.

Операція змінює залишок.

Цей принцип є фундаментом усієї системи.

## StockDocument

Універсальний документ руху майна. Тип `TRANSFER` передає майно між двома МВО,
тип `ISSUE` видає майно зовнішньому одержувачу. Статуси: `DRAFT`, `POSTED`,
`CANCELLED`. Рядки документа зберігаються у `StockDocumentLine`.
