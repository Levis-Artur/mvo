# System Architecture

Version: 1.0 (Draft)

---

# Purpose

Цей документ описує архітектуру системи MVO Inventory Management System.

Його мета — забезпечити єдине розуміння структури проєкту для всіх розробників.

Будь-які значні архітектурні зміни повинні бути відображені в цьому документі.

---

# High-Level Architecture

Система побудована за класичною трирівневою архітектурою.

```
Browser
      │
      ▼
Frontend (Next.js)
      │
REST API
      ▼
Backend (NestJS)
      │
Prisma ORM
      ▼
PostgreSQL
```

---

# Architecture Principles

Проєкт побудований на наступних принципах:

- Separation of Concerns
- Single Responsibility Principle
- Modular Architecture
- Type Safety
- Backend First
- Stateless API
- Explicit Validation

# Frontend

Frontend відповідає виключно за:

- відображення інформації;
- взаємодію з користувачем;
- локальний UI State;
- виклики REST API.

Frontend не приймає рішень щодо безпеки.

---

# Backend

Backend є єдиним джерелом істини.

Саме backend:

- перевіряє права;
- виконує бізнес-логіку;
- працює з базою даних;
- веде аудит;
- перевіряє цілісність даних.

---

# Database

PostgreSQL використовується як основне сховище даних.

Всі зміни проходять через Prisma ORM.

Прямий SQL допускається лише у випадках, коли Prisma не підтримує необхідну функціональність.

---

# ORM

Для доступу до БД використовується Prisma.

Усі моделі повинні бути описані у schema.prisma.

Будь-які зміни структури БД виконуються лише через Prisma Migrations.

---

# API

Frontend взаємодіє лише через REST API.

Заборонено:

- прямий доступ до БД;
- використання прихованих endpoint;
- дублювання бізнес-логіки на frontend.

---

# Authentication

Авторизація повністю виконується backend.

Frontend лише зберігає отриману сесію.

Перевірка доступу виконується виключно сервером.

---

# Authorization

Кожен endpoint визначає:

- хто має право виконати операцію;
- над якими об'єктами;
- за яких умов.

---

# Logging

Усі критичні події логуються.

Логи повинні містити:

- timestamp;
- requestId;
- userId;
- operation;
- результат.

---

# Transactions

Операції, що змінюють дані, повинні використовувати транзакції Prisma.

Часткове оновлення даних не допускається.

---

# Error Handling

Backend використовує єдину модель помилок.

Усі помилки повертаються у стандартизованому форматі.

Stack trace ніколи не повертається клієнту.

---

# Validation

Усі DTO проходять серверну валідацію.

Frontend може виконувати додаткову перевірку лише для покращення UX.

# Scalability

Архітектура повинна дозволяти:

- горизонтальне масштабування;
- винесення окремих сервісів;
- інтеграцію з іншими системами;
- додавання нових модулів без переписування існуючих.

---

# Architecture Rules

Не допускається:

- бізнес-логіка у React-компонентах;
- SQL у контролерах;
- доступ до БД з frontend;
- дублювання бізнес-логіки;
- обхід системи ролей.
